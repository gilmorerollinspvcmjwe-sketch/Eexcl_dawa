# 三消模块需求规格

## Why
在当前 Excel 风格游戏中心内，高还原交付经典 Match-3 / Bejeweled / 开心消消乐系核心体验，提供与现有 PvZ（防线经营）、Snake（路径生存）、Tetris（堆叠整理）不同的"短反馈+连锁爽感"玩法，满足玩家对短决策、连锁反馈、目标解题的需求。

## What Changes
- 新增 Sheet16（三消主玩法）：选关、开局、战斗、HUD、结算
- 新增 Sheet17（三消图鉴/训练）：特殊块说明、障碍规则、组合实验场、练习关入口
- 新增三消核心运行时：棋盘系统、交换判定、消除判定、掉落与连锁、特殊块体系、障碍体系
- 新增三消关卡数据层：8个内容包、100+关卡脚本、难度曲线参数表
- 新增三消存档系统：关卡进度、星级、最佳分数、每日挑战记录

## Impact
- Affected specs: 新增独立模块，不影响现有模块
- Affected code: 新增 `src/features/match3/` 目录、`src/components/match3/` 目录、`src/styles/match3.css`
- Sheet 结构: Sheet16/Sheet17 新增，Sheet1 Hub 新增三消入口卡

## ADDED Requirements

### Requirement: 核心玩法机制

#### 交换规则
- 系统应仅允许曼哈顿相邻交换（上下左右）
- 系统应在非法交换时执行动画回弹，不扣步数
- 系统应在合法交换（至少触发一个消除群组）时提交操作并扣步数（步数关）
- 系统应保证输入优先级：swap > resolve > inputLock
- 系统应在结算中禁二次输入，防状态穿透

#### 棋盘与元素规格
- 系统应支持基础棋盘 8x8（教程关可用 7x7，高压关可用 9x9）
- 系统应支持基础颜色 5~6 种（首发建议 6 种）
- 系统应保证每局存在合法交换；若无合法交换则自动洗牌（保留障碍状态）

#### 判定、掉落与连锁
- 系统应同步扫描横纵连续段，标记所有待消格
- 系统应支持一次结算同时存在多个群组，统一清除后再掉落
- 系统应按从下到上逐列压缩掉落，空位由顶部补牌
- 系统应定义连锁：同一次玩家操作触发的第N次自动消除即第N段连锁
- 系统应实现连锁倍率：1.0 -> 1.25 -> 1.5 -> 1.75 -> 2.0 上限

#### 胜利/失败条件
- 系统应在达成关卡目标时判定胜利
- 系统应在步数耗尽且目标未达成时判定失败（步数关）
- 系统应在时间耗尽且目标未达成时判定失败（计时模式）
- 系统应在关键目标未完成时判定失败（如收集物未到达底部）
- 系统应提供清晰失败原因和改进建议

#### Scenario: 三消核心玩法闭环
- **WHEN** 玩家进入 Sheet16 并选择关卡
- **THEN** 系统加载棋盘、颜色池、目标和障碍规则
- **WHEN** 玩家执行相邻交换
- **THEN** 系统判定是否合法交换
- **WHEN** 交换形成消除群组
- **THEN** 系统触发消除、掉落和连锁判定
- **WHEN** 达成关卡目标
- **THEN** 系统判定胜利并显示结算面板
- **WHEN** 步数/时间耗尽且目标未达成
- **THEN** 系统判定失败并显示结算面板

### Requirement: 游戏界面元素

#### HUD布局（不遮挡棋盘，最多两行信息）
- 系统应在第一行显示：模式、关卡名、目标摘要、剩余步数/时间
- 系统应在第二行显示：分数、连锁段数、特殊块任务进度、障碍威胁
- 系统应在右侧提供：暂停、重开、加速动画按钮（仅表现层，不影响逻辑）

#### 棋盘布局
- 系统应渲染棋盘网格（8x8为主）
- 系统应渲染普通色块（6种颜色）
- 系统应渲染特殊块（条纹、包装、彩球）
- 系统应渲染障碍（冻结层、锁链块、木箱/石块、传送口、蔓延块）
- 系统应渲染目标收集物（掉落收集关）

#### 结算面板
- 系统应在胜利时显示：星级、分数明细、连锁峰值、关键目标完成率、推荐下一包
- 系统应在失败时显示：未完成目标清单、失败主因（步数不足/障碍失控/时间耗尽）、建议策略
- 系统应提供：重开本局、返回选关、继续训练三个出口

#### Excel风格要求
- 系统应保持工作簿风格：公式栏显示当前状态摘要
- 系统应保持状态栏显示连锁/奖励提示
- 系统应保持Sheet标签切换功能

#### Scenario: HUD信息完整性
- **WHEN** 玩家进入战斗状态
- **THEN** 系统在顶部显示模式、关卡、目标、剩余步数/时间
- **WHEN** 触发连锁消除
- **THEN** 系统在状态栏显示连锁段数和倍率
- **WHEN** 障碍威胁增加
- **THEN** 系统在第二行显示障碍威胁状态

### Requirement: 关卡设计规则

#### 难度梯度（4段）
- 系统应支持前10关：单目标+低障碍，建立交换与特殊块直觉
- 系统应支持11~30关：引入复合目标和一层障碍，要求规划步数
- 系统应支持31~60关：障碍组合+掉落目标+限步压力，逼迫连锁效率
- 系统应支持61+关：高密障碍与特殊块联动关，强调"先解场，再冲目标"

#### 关卡目标类型
- 系统应支持分数达标目标（Score Target）
- 系统应支持颜色收集目标（Collect Colors）
- 系统应支持清理覆盖层目标（Clear Jelly / Frost）
- 系统应支持掉落收集目标（Drop Ingredients）
- 系统应支持障碍清除目标（Break Crates / Chains / Stones）
- 系统应支持复合目标（如"20步内清理果冻并收集10红色块")

#### 关卡脚本结构
- 系统应定义每关：boardTemplateId、palette（颜色集合）
- 系统应定义每关：moveLimit或timeLimitMs
- 系统应定义每关：targets[]（类型、数量、优先级）
- 系统应定义每关：blockers[]（类型、初始坐标、层数）
- 系统应定义每关：spawnRules（补牌概率、特殊块注入策略）
- 系统应定义每关：tutorialHint（可选）

#### 难度控制旋钮
- 系统应支持颜色种类数量调节（5/6/7）
- 系统应支持初始合法交换数量阈值调节
- 系统应支持障碍密度与层数调节
- 系统应支持目标数与目标组合复杂度调节
- 系统应支持步数预算调节（目标理论最优步数的1.3~2.2倍）
- 系统应支持关键颜色生成权重调节（防止"目标色枯竭")

#### Scenario: 关卡难度递进
- **WHEN** 玩家完成前10关
- **THEN** 系统解锁11~30关并引入复合目标
- **WHEN** 玩家完成11~30关
- **THEN** 系统解锁31~60关并增加障碍组合
- **WHEN** 玩家完成31~60关
- **THEN** 系统解锁61+关并启用高密障碍

### Requirement: 角色/道具系统（特殊块体系）

#### 特殊块生成规则
- 系统应在四连（横/纵）时生成条纹块（行清/列清）
- 系统应在五连直线时生成彩球（颜色炸弹）
- 系统应在T/L形五连时生成包装块（3x3爆炸）

#### 特殊块组合规则（首发必须）
- 系统应支持条纹+条纹：十字双清
- 系统应支持条纹+包装：三行三列爆破
- 系统应支持包装+包装：大范围双爆
- 系统应支持彩球+普通色块：清屏该颜色
- 系统应支持彩球+特殊块：全色转化为对应特殊块并触发

#### 特殊块属性
- 系统应定义特殊块类型：条纹（横/纵）、包装、彩球
- 系统应定义特殊块触发范围和效果
- 系统应定义特殊块得分倍率

#### Scenario: 特殊块组合效果
- **WHEN** 玩家交换条纹块与条纹块
- **THEN** 系统触发十字双清效果
- **WHEN** 玩家交换彩球与普通色块
- **THEN** 系统清屏该颜色所有块
- **WHEN** 玩家交换彩球与条纹块
- **THEN** 系统将全色转化为条纹块并触发

### Requirement: 障碍体系

#### 障碍类型（首发建议）
- 系统应支持冻结层：需1~2次消除击破
- 系统应支持锁链块：被锁格不可移动，消除邻接后解锁
- 系统应支持木箱/石块：不可交换，不可下落，需邻接爆破
- 系统应支持传送口：入口/出口映射，形成非线性掉落路径
- 系统应支持巧克力/蔓延块：每回合扩散，逼迫玩家主动处理

#### 障碍规则
- 系统应定义障碍击破条件和层数
- 系统应定义障碍对交换和掉落的影响
- 系统应定义蔓延块的扩散规则和阈值
- 系统应提供障碍优先级提示

#### Scenario: 障碍处理效果
- **WHEN** 玩家消除冻结层邻接块
- **THEN** 系统减少冻结层层数
- **WHEN** 冻结层层数归零
- **THEN** 系统清除冻结层
- **WHEN** 蔓延块每回合扩散
- **THEN** 系统增加蔓延块覆盖范围

### Requirement: 性能优化指标

#### 帧率要求
- 系统应保持目标60fps稳定运行
- 系统应在常规棋盘（8x8）保持流畅
- 系统应在高压棋盘（9x9）保持流畅

#### 资源占用限制
- 系统应控制单关卡内存占用 < 30MB
- 系统应控制动画渲染不阻塞主线程
- 系统应使用事件队列播放动画，避免逻辑帧阻塞

#### 动画与逻辑解耦
- 系统应先结算逻辑帧，动画通过事件队列播放
- 系统应支持加速动画（仅表现层，不影响逻辑）

#### Scenario: 性能达标验证
- **WHEN** 常规棋盘运行时
- **THEN** 系统保持60fps帧率
- **WHEN** 触发连锁消除
- **THEN** 系统动画不阻塞逻辑帧
- **WHEN** 高压棋盘运行时
- **THEN** 系统保持不低于45fps帧率

### Requirement: 内容包体系

#### 8个正式内容包
1. 新手入门包：三消与交换基础（12关）
2. 步数策略包：有限步目标闯关（20关）
3. 果冻清理包：覆盖层拆除（18关）
4. 下落收集包：物件护送到底（16关）
5. 障碍拆解包：锁链、木箱与蔓延压制（24关）
6. 特效连锁包：特殊块组合大师（14关）
7. 限时冲分包：Blitz与连击速刷（3档时长+6张主题盘）
8. 每日挑战与赛季回流包（首发30个模板轮换）

#### 内容包规则
- 系统应支持内容包独立接入，无需改核心状态机
- 系统应支持内容包增量上线
- 系统应首发优先交付：新手入门包 + 步数策略包 + 果冻清理包

#### Scenario: 内容包接入

### Requirement: 技术架构边界

#### 架构原则
- 系统应分离状态与渲染：规则在simulation层，UI只消费状态并发动作
- 系统应统一输入动作：swap、pause、restart、toggle-speed
- 系统应支持可存档状态序列化：仅保存逻辑状态，不保存渲染对象
- 系统应分离DOM HUD与棋盘渲染：棋盘可用DOM/CSS Grid（首发），HUD/菜单保持DOM层

#### 模块拆分
- 系统应创建：match3Types.ts、match3LevelCatalog.ts、match3BoardState.ts、match3ProgressStorage.ts
- 系统应创建组件：Match3Sheet.tsx、Match3Board.tsx、Match3Hud.tsx、Match3ResultPanel.tsx、Match3LabSheet.tsx、match3.css

#### 关键状态机
- 系统应实现状态机：setup -> playing -> resolving -> won/lost
- 系统应实现resolving子阶段：markMatches -> clear -> drop -> spawn -> checkCascade
- 系统应仅在playing状态可用输入，resolving期间锁定

#### 关卡数据化边界
- 系统应保证关卡定义纯数据（JSON/TS对象），禁止把关卡特例写进组件
- 系统应保证障碍、目标、掉落规则全部走catalog配置
- 系统应保证每个内容包对应独立level group

#### 测试边界
- 系统应测试：合法/非法交换判定、匹配扫描准确性（含交叉群组）、掉落与补牌稳定性、特殊块生成与组合结果、洗牌后可行动作保证
- 系统应测试：关卡胜负判定、连锁倍率积分、进度存储与读取

#### Scenario: 架构分层验证
- **WHEN** 运行三消模块
- **THEN** Simulation层与Render层保持分离
- **WHEN** 触发游戏事件
- **THEN** 事件流正确传递到HUD和音效系统
- **WHEN** 进入resolving状态
- **THEN** 输入锁定，防止竞态
