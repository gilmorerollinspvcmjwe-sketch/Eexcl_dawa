# 吃豆人模块需求规格

## Why
在现有 Excel 风格游戏中心中高还原 Pac-Man 原版体验，提供"固定迷宫+路线学习+四鬼差异AI+模式节奏切换"玩法，满足玩家对规则确定、路线可学、风险可控、节奏不断加压的需求，与现有模块形成节奏互补。

## What Changes
- 新增 Sheet12（吃豆人主玩法）：选关、模式设置、正式对局、HUD、结算
- 新增 Sheet13（吃豆人图鉴与练习）：鬼魂图鉴、模式说明、路径练习与回放诊断
- 新增吃豆人核心运行时：迷宫系统、豆子系统、能量豆系统、水果系统、四鬼AI系统、模式时序系统
- 新增吃豆人关卡数据层：8个内容包、21+循环关、难度曲线参数表
- 新增吃豆人存档系统：关卡进度、最佳分数、最快清图、无伤记录

## Impact
- Affected specs: 新增独立模块，不影响现有模块
- Affected code: 新增 `src/features/pacman/` 目录、`src/components/pacman/` 目录、`src/styles/pacman.css`
- Sheet 结构: Sheet12/Sheet13 新增，Sheet1 Hub 新增吃豆人入口卡

## ADDED Requirements

### Requirement: 核心玩法机制

#### 输入模型
- 系统应支持方向输入：Arrow / WASD
- 系统应支持动作输入：Pause、Restart、Exit
- 系统应缓存"下一方向"，在进入新格前判定可转向时执行
- 系统应保证手感接近原版（转向缓存机制）

#### 核心循环
- 系统应读取输入，更新Pac-Man方向与位移
- 系统应处理豆子、能量豆、水果拾取与得分
- 系统应更新四鬼状态机（House / Scatter / Chase / Frightened / Eaten / Respawn）
- 系统应进行碰撞判定（Pac-Man vs Ghost / Wall / Tunnel）
- 系统应结算当前帧事件（加分、掉命、清图、进关、音效触发）
- 系统应推送HUD与状态栏摘要

#### 地图、豆子、能量豆、传送门
- 系统应定义地图单元类型：wall / path / pellet / energizer / ghostHouse / tunnel / fruitSpawn
- 系统应支持标准迷宫：首发至少包含原版经典迷宫1张，并支持后续多迷宫目录
- 系统应定义豆子总量：按地图定义（经典图约240粒），不写死在代码常量
- 系统应定义能量豆：固定在4个角位，触发Frightened全局状态窗口
- 系统应定义传送门：左右通道联通，鬼在tunnel中应用速度修正

#### 水果系统
- 系统应按豆子剩余阈值触发水果（建议2次触发窗口）
- 系统应使用原版风格进阶水果序列（樱桃、草莓、橙子、苹果、甜瓜、银河舰、铃铛、钥匙）
- 系统应限制水果出现时间，提供"路线压缩决策"

#### 胜利/失败条件
- 系统应在当前迷宫所有普通豆清空时判定关卡胜利
- 系统应在生命值归零时判定关卡失败
- 系统应支持章节通关：按章节定义完成若干关或达成目标

#### Scenario: 吃豆人核心玩法闭环
- **WHEN** 玩家进入Sheet12并选择关卡
- **THEN** 系统加载迷宫、豆子、能量豆、水果规则和四鬼初始状态
- **WHEN** 玩家输入方向
- **THEN** 系统缓存下一方向并在可转向时执行
- **WHEN** Pac-Man吃到豆子
- **THEN** 系统加分并更新剩余豆子数
- **WHEN** Pac-Man吃到能量豆
- **THEN** 系统触发Frightened全局状态窗口
- **WHEN** Pac-Man吃到水果
- **THEN** 系统加分并记录水果收益
- **WHEN** Pac-Man与鬼碰撞（非Frightened状态）
- **THEN** 系统扣命并触发重生
- **WHEN** 清空所有普通豆
- **THEN** 系统判定胜利并显示结算面板
- **WHEN** 生命值归零
- **THEN** 系统判定失败并显示结算面板

### Requirement: 游戏界面元素

#### HUD布局（战斗页）
- 系统应显示：当前分数、当前关卡、剩余生命
- 系统应显示：当前模式（Scatter/Chase/Frightened）
- 系统应显示：剩余豆子数、水果状态（未出/出现中/已拾取）
- 系统应可选显示：当前鬼速度倍率、Frightened剩余时间、当前关卡参数档位

#### 棋盘/迷宫布局
- 系统应渲染迷宫墙壁和通道
- 系统应渲染普通豆子（小点）
- 系统应渲染能量豆（大点，4个角位）
- 系统应渲染Pac-Man（带方向动画）
- 系统应渲染四鬼（Blinky红、Pinky粉、Inky青、Clyde橙）
- 系统应渲染传送门（左右通道）
- 系统应渲染水果（出现时）

#### 结算面板
- 系统应在胜利时显示：用时、剩余命、豆子清空率、水果收益、吃鬼次数
- 系统应在失败时显示：死亡位置热区、最后10秒输入轨迹、建议改进点
- 系统应提供：重开本局、返回选关、进入练习三个出口

#### Excel风格要求
- 系统应保持工作簿风格：公式栏显示当前状态摘要
- 系统应保持状态栏显示模式/奖励提示
- 系统应保持Sheet标签切换功能

#### Scenario: HUD信息完整性
- **WHEN** 玩家进入战斗状态
- **THEN** 系统显示分数、关卡、生命、模式、豆子数
- **WHEN** 能量豆触发Frightened状态
- **THEN** 系统显示Frightened剩余时间
- **WHEN** 水果出现
- **THEN** 系统显示水果状态为"出现中"

### Requirement: 关卡设计规则

#### 关卡阶段建议（对应原版学习曲线）
- 系统应支持Level 1-2：惊吓窗口长，允许学习鬼行为
- 系统应支持Level 3-5：追击速度提升，强制优化路线
- 系统应支持Level 6-9：惊吓窗口变短，反打价值变成"窗口技巧"
- 系统应支持Level 10-16：更强调避让与路径预判
- 系统应支持Level 17+：接近原版高压状态，主要靠路线稳定性过图

#### 速度与时序表驱动
- 系统应使用levelTuningTable做参数表，不把速度和惊吓时长写死在逻辑分支中
- 系统应定义关键字段：pacmanSpeed、ghostSpeed、tunnelGhostSpeedMultiplier
- 系统应定义关键字段：frightenedDurationMs、frightenedBlinkMs
- 系统应定义关键字段：scatterChaseSchedule、fruitValueSet

#### 还原优先级
- 系统应优先实现P0：鬼目标逻辑、模式时序、清图胜利、能量豆反打、生命与重生
- 系统应其次实现P1：水果触发序列、鬼屋精细放行、鬼速度细分（含tunnel修正）
- 系统应最后实现P2：额外迷宫、镜像变体、训练回放、挑战脚本编辑器

#### Scenario: 关卡难度递进
- **WHEN** 玩家完成Level 1-2
- **THEN** 系统解锁Level 3-5并提升追击速度
- **WHEN** 玩家完成Level 3-5
- **THEN** 系统解锁Level 6-9并缩短惊吓窗口
- **WHEN** 玩家完成Level 6-9
- **THEN** 系统解锁Level 10-16并强调避让与路径预判

### Requirement: 角色/道具系统（四鬼AI）

#### 四鬼目标逻辑（高还原）
- 系统应实现Blinky（红）：直接追踪Pac-Man当前格目标
- 系统应实现Pinky（粉）：瞄准Pac-Man前方若干格（含原版"向上时偏移异常"可配置开关）
- 系统应实现Inky（青）：以Blinky与Pac-Man前方点构造向量目标，形成夹击
- 系统应实现Clyde（橙）：远距离追击，近距离回角落，形成节奏断点

#### 全局模式时序
- 系统应按"Scatter/Chase周期表"驱动模式切换，不用纯随机切换
- 系统应在切换时鬼执行一次反向，保证玩家可读
- 系统应在Frightened结束后恢复原模式节奏，不重置整局时序

#### 鬼魂状态机
- 系统应实现鬼魂状态：House（鬼屋内）、Scatter（散开）、Chase（追击）、Frightened（惊吓）、Eaten（被吃）、Respawn（重生）
- 系统应实现状态切换逻辑和动画

#### 鬼屋放行机制
- 系统应采用"豆子计数+时间兜底"的混合放行规则
- 系统应不同鬼放行门槛独立配置，确保开局压力可控且有学习价值

#### Scenario: 四鬼AI行为差异
- **WHEN** Blinky追击
- **THEN** 系统直接追踪Pac-Man当前格
- **WHEN** Pinky追击
- **THEN** 系统瞄准Pac-Man前方若干格
- **WHEN** Inky追击
- **THEN** 系统以Blinky与Pac-Man前方点构造向量目标
- **WHEN** Clyde追击
- **THEN** 系统远距离追击，近距离回角落

#### 动画效果

- 系统应保持公式栏和状态栏风格一致

### Requirement: 性能优化指标

#### 帧率要求
- 系统应保持目标60fps稳定运行
- 系统应使用固定步进（建议60Hz逻辑步）
- 系统应渲染可跟随浏览器帧率

#### 资源占用限制
- 系统应控制单关卡内存占用 < 30MB
- 系统应控制动画渲染不阻塞主线程

#### Simulation与Rendering边界
- 系统应分离Simulation层（网格坐标、移动状态机、鬼AI、碰撞、计分、关卡推进）
- 系统应分离Rendering层（DOM/CSS渲染迷宫、角色、豆子、UI动画，不持有真状态）

#### Scenario: 性能达标验证
- **WHEN** 运行吃豆人模块
- **THEN** 系统保持60fps帧率
- **WHEN** 四鬼同时追击
- **THEN** 系统保持流畅
- **WHEN** 触发Frightened状态
- **THEN** 系统动画不阻塞逻辑帧

### Requirement: 内容包体系

#### 8个正式内容包
1. 经典街机包：完整交付Pac-Man核心体验（21+循环关）
2. 路线教学包：让新玩家学会安全路线+收豆顺序+能量豆时机（10关）
3. 四鬼认知包：把四鬼差异从"感觉"变成可观察规则（12关）
4. 水果冲分包：强调风险换收益的高分路线（8关）
5. 限命挑战包：面向熟练玩家的高压稳定性测试（5章）
6. 速通竞赛包：把清图效率变成主目标（15关）
7. 迷宫变体包：在不改核心规则下扩大内容寿命（20关）
8. 无尽生存包：长期回流内容（无上限）

#### 内容包规则
- 系统应支持内容包独立接入，无需改核心状态机
- 系统应支持内容包增量上线
- 系统应首发优先交付：经典街机包 + 路线教学包

#### Scenario: 内容包接入
- **WHEN** 新增内容包数据
- **THEN** 系统自动接入关卡目录
- **WHEN** 玩家选择内容包
- **THEN** 系统加载对应关卡脚本和规则

### Requirement: 技术架构边界

#### 实现路径选择
- 系统应沿用现有React+逻辑状态层架构，不强依赖Phaser
- 系统应复用现有Snake/Tetris/PvZ同类模块边界

#### 模块拆分
- 系统应创建：pacmanTypes.ts、pacmanBoardState.ts、pacmanAi.ts、pacmanMapRegistry.ts、pacmanLevelTuning.ts、pacmanContent.ts、pacmanSelectors.ts、pacmanStorage.ts
- 系统应创建组件：PacmanSheet.tsx、PacmanBoard.tsx、PacmanHud.tsx、PacmanOverlay.tsx、PacmanGuideSheet.tsx（可延期）

#### Simulation与Rendering边界
- 系统应分离Simulation层（网格坐标、移动状态机、鬼AI、碰撞、计分、关卡推进）
- 系统应分离Rendering层（DOM/CSS渲染迷宫、角色、豆子、UI动画，不持有真状态）
- 系统应使用固定步进（建议60Hz逻辑步），渲染可跟随浏览器帧率

#### 数据与存档边界
- 系统应仅存可序列化状态，不存DOM运行态
- 系统应存档分为：settings、progress、lastRunStats
- 系统应所有关卡参数走内容表，避免逻辑分支硬编码

#### 测试边界
- 系统应测试：路径阻挡与转向缓存判定、四鬼目标计算正确性（含Pinky特例开关）、模式时序切换与反向行为、Frightened触发/结束与吃鬼计分递增、清图进关与失败结算、水果触发阈值与超时消失

#### Scenario: 架构分层验证
- **WHEN** 运行吃豆人模块
- **THEN** Simulation层与Render层保持分离
- **WHEN** 触发游戏事件
- **THEN** 事件流正确传递到HUD和音效系统
- **WHEN** 存档
- **THEN** 仅存可序列化状态，不存DOM运行态
