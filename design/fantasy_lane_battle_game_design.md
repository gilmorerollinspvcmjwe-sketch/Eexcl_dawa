# 奇幻战线对推（Fantasy Lane Battle）Excel 游戏详细设计

- 日期：2026-04-14
- 项目：`excel-aim-trainer`
- 设计目标：在现有 Excel 风格游戏合辑中新增一款“赤潮式单线对推”模块，用传统奇幻题材包装成“轻配队 + 局内出兵 + 手动技能 + 英雄局外成长”的 PvE 闯关玩法
- 设计阶段：实施前详细设计

---

## 1. 游戏概述

### 1.1 一句话介绍

奇幻战线对推是一个“带着自己预设好的兵种池，在一条横版战线上不断攒费出兵、释放技能、把敌军一路推回城门”的策略游戏。

### 1.2 核心卖点

- **赤潮骨架，但更轻更好懂**：保留配队、出兵、克制、战线推进的乐趣，不做重度 PvP 和复杂微操
- **传统奇幻题材天然顺手**：哥布林、兽人、法师、狮鹫、火龙这些单位，玩家一眼就知道谁是前排、谁是输出、谁是大怪
- **短局节奏适合 arcade 合集**：单局控制在 5 到 8 分钟，理解快、失败重开也不累
- **局内外都有成长感**：局内拼出兵节奏和技能时机，局外养英雄和解锁兵种
- **Excel 壳层容易伪装**：战线可以包装成“流程推进条”，兵种可以包装成“执行队列”，技能可以包装成“紧急指令”

---

## 2. 原版玩法分析与本作取舍

### 2.1 赤潮式玩法的核心机制

| 机制 | 原始含义 | 本作保留方式 |
| --- | --- | --- |
| 开局配队 | 先带一套可用兵种进入战斗 | 保留，改为轻配队，首发每局带 8~10 个兵种 |
| 局内出兵 | 持续攒费，按节奏出兵 | 保留，作为主操作 |
| 自动行军交战 | 单位自动前进与攻击 | 保留，不做单位手操 |
| 兵种克制 | 前排、后排、空军、群伤之间的相互制衡 | 保留，首发只做清晰标签，不堆过深公式 |
| 战线推进 | 存活部队继续前推并威胁主堡 | 保留，作为关卡胜负核心 |
| 强策略博弈 | 更偏 PvP 读对手 | 缩减，改为 PvE 读关卡主题和敌军脚本 |

### 2.2 本作的关键改动

- 把重 PvP 改为 **单人章节 PvE**
- 把重度编队改为 **轻配队**
- 在局内加入 **玩家手动技能**
- 把英雄改为 **局外成长核心**，不做场上手操角色
- 单局压缩到 **5~8 分钟**，更符合本项目的短局结构

### 2.3 核心体验目标

- 玩家看一眼就能知道当前战线是“我顶住了”还是“要崩了”
- 玩家能从单位标签和体型上立刻理解克制关系
- 玩家能通过一次正确的技能释放完成保线或反推
- 玩家能明显感到英雄升级后，整套玩法越来越顺手

## 3. 核心玩法设计

### 3.1 基本循环

```text
进入关卡
  -> 选择英雄、确认本局兵种池
  -> 战斗开始，金币持续增长
  -> 玩家根据前线局势不断出兵
  -> 英雄技能和战术技能在关键时机手动释放
  -> 敌方按关卡脚本持续出兵和放技能
  -> 存活部队继续推进并攻击敌方主堡
  -> 击破敌方主堡 / 关底 Boss = 胜利
  -> 我方主堡被推穿 = 失败
```

### 3.2 一局中的主要操作

- **出兵**：玩家的主操作，决定前排、输出、对空和高费怪何时登场
- **攒费**：决定是稳线过渡，还是憋大波总攻
- **放技能**：解决快崩线、清杂兵或强行突破的关键节点
- **观察战线**：根据当前敌军主题调整下一轮出兵

### 3.3 单局节奏

- 开局 30~60 秒：以试探和稳线为主，帮助玩家读懂敌军关卡主题
- 中段 2~4 分钟：兵种克制和资源节奏开始发力，进入主对推阶段
- 尾段 1~2 分钟：Boss、飞龙群或高费重压登场，需要技能和高费兵协同收尾

## 4. 战场规则设计

### 4.1 战场结构

采用 **单线横版战场**，但把单位站位抽象成四层，方便读图和后续实现：

- **前列**：肉盾、近战单位
- **中列**：短射程、法术和突击单位
- **后列**：弓手、投手、召唤师
- **空层**：飞行单位单独结算

这样既保留了单线对推的清晰感，也能让空军和地面部队同时存在。

### 4.2 推进规则

- 双方单位从各自主堡持续出发
- 单位遇到攻击目标时自动停下交战
- 一侧清空前方阻挡后会继续推进
- 单位进入敌方主堡攻击区后会开始拆堡
- 敌方主堡血量归零则关卡胜利；我方主堡血量归零则关卡失败

### 4.3 资源规则

- 金币按固定节奏持续回复
- 击杀精英怪或通过小波段会给额外金币
- 高费单位价格高、冷却长，是后期节奏核心
- 玩家不能囤积无限资源，建议设置软上限，防止完全存钱不出兵

### 4.4 出兵规则

- 每个兵种有独立费用和冷却
- 玩家可快速连续点选多个兵种，形成“前排 + 输出 + 对空”的组合波次
- 兵种一旦出场即自动行军和作战
- 同类兵种可以连续出，但会被费用和冷却限制

## 5. 兵种系统设计

### 5.1 兵种职责分层

首发兵种不靠复杂名字建立深度，而是靠 **职责清晰** 建立深度。

| 类别 | 代表单位 | 主要作用 |
| --- | --- | --- |
| 前排肉盾 | 哥布林盾兵、兽人重甲兵、石像卫士 | 顶线、吸收伤害、保护后排 |
| 近战冲锋 | 狼骑兵、狂战士、地穴爬兽 | 撕开阵型、快速压线 |
| 远程持续输出 | 弓箭手、精灵射手、火枪佣兵 | 稳定输出，是中后期主力火力 |
| 法术/范围兵 | 火焰术士、冰霜女巫、瘟疫投手 | 清理杂兵海、处理密集阵型 |
| 空中/特殊兵 | 狮鹫骑士、蝙蝠群、幼龙 | 跨层打击、迫使玩家补对空 |
| 高费终结兵 | 树人古卫、食人魔领主、火龙 | 一波改变战线，作为总攻核心 |

### 5.2 首发建议兵种池

首发可设计 18~24 个兵种，供玩家局外从中挑选 8~10 个带入关卡。

建议首发分布：

- 前排肉盾：4 个
- 近战冲锋：4 个
- 远程持续输出：4 个
- 法术/范围兵：4 个
- 空中/特殊兵：3 个
- 高费终结兵：3 个

### 5.3 兵种关键字段

每个兵种只保留少数高价值字段，避免设计过厚：

- 费用
- 生命
- 攻击
- 攻击距离
- 移动速度
- 攻击间隔
- 单位体型
- 标签

### 5.4 首发标签体系

建议首发只做以下标签：

- `地面`
- `空中`
- `轻甲`
- `重甲`
- `群攻`
- `对空`
- `破甲`
- `召唤`
- `治疗`
- `Boss`

这些标签足够支撑第一版策略判断：

- 对面飞龙多，要补对空
- 对面重甲多，要补破甲
- 对面杂兵海多，要补群伤
- 我方前排不够，要先补肉盾再谈输出

## 6. 英雄与技能系统设计

### 6.1 英雄定位

英雄不作为场上直接手操单位，而是本局的 **成长核心 + 技能来源**。

局内作用：

- 提供阵容方向性被动
- 提供一个英雄主动技能

局外作用：

- 升级
- 解锁分支强化
- 强化主动技能

### 6.2 首发英雄建议

#### 战帅

- 方向：前排和冲锋
- 被动：近战单位生命提升，首次接敌获得短暂增益
- 主动技能：战吼，短时间提高我方推进速度和伤害

#### 法导师

- 方向：法术与范围清场
- 被动：法系单位伤害提高，技能冷却略缩短
- 主动技能：陨火/冰暴，对一片区域造成范围伤害或冻结

#### 驭龙者

- 方向：空军和高费大怪
- 被动：空军或高费单位登场时获得短时间强化
- 主动技能：龙焰扫射，对战线一段区域持续灼烧

### 6.3 英雄成长结构

首发采用简单的三段式成长：

- **等级成长**：基础属性或被动增强
- **分支强化**：二选一强化方向，偏某种打法
- **技能强化**：主动技能范围、持续时间或附加效果提升

这样玩家能清楚理解“我升级这个英雄后，打法会更偏哪边”。

### 6.4 玩家技能结构

每局可带：

- `1 个英雄主动技能`
- `1 个通用战术技能`

通用技能建议从下面几类中选择：

- 火球：清杂兵
- 治疗：保前线
- 加速：打一波总攻
- 冰冻：拖住 Boss 或高费推进
- 护盾：保关键单位

### 6.5 技能设计原则

- 冷却偏长，不能无脑连按
- 视觉和结果都要明显
- 重点服务“保线”和“一波推穿”
- 不抢出兵系统的主地位

## 7. 关卡与章节设计

### 7.1 章节结构

建议采用 **5 章 x 6 关 = 30 关** 的首发规模。

推荐章节主题：

1. 哥布林边境
2. 亡骨荒原
3. 冰霜峡谷
4. 熔火山口
5. 龙火王庭

### 7.2 每章关卡节奏

- 第 1 关：新主题导入
- 第 2 关：加入变体敌军
- 第 3 关：提高出兵压力
- 第 4 关：引导玩家使用技能或高费兵
- 第 5 关：强化本章主题
- 第 6 关：Boss 关

### 7.3 敌军主题设计

每关最好有清晰的“考点”，而不是单纯数值变高。

建议首发主题：

- 哥布林海：考验群伤
- 重甲推进：考验破甲
- 飞龙压顶：考验对空
- 召唤狂潮：考验清场和稳线
- 法术轰炸：考验技能时机和保线能力
- Boss 压阵：考验高费兵和总攻节奏

### 7.4 Boss 设计原则

- Boss 不只是血厚，而是会改变战场规则
- 每个 Boss 至少有一个鲜明技能
- Boss 必须逼玩家用到英雄技能或战术技能

示例：

- 骸骨巨王：会召唤墓地小怪，逼玩家补范围伤害
- 冰霜巨龙：持续冻住前线，逼玩家调整出兵节奏
- 熔岩暴君：大范围灼烧，逼玩家快速拆掉前排压力

## 8. Excel 世界观包装

### 8.1 包装方向

外层视觉和交互继续保留 Excel 风格，但游戏世界观本体使用 **传统奇幻 RPG** 表达。

也就是说：

- 玩家看到的单位是哥布林、兽人、法师、火龙
- 但外层容器仍像一个“作战调度表”

### 8.2 Excel 壳层映射

| 奇幻概念 | Excel 壳层说法 |
| --- | --- |
| 主堡生命 | 核心进度/据点耐久 |
| 出兵按钮 | 执行队列 |
| 英雄技能 | 紧急指令 |
| 战术技能 | 辅助命令 |
| 战线推进 | 推进进度条 |
| 兵种池 | 预设编组 |
| 关卡章节 | 战役页签 |

### 8.3 为什么这样包装

- 保持现有项目统一风格
- 不破坏奇幻单位本身的直觉辨识
- 让 UI 仍然能自然挂在工作簿、表格、状态栏里

## 10. 美术与音效方向

### 10.1 美术方向

- 主体单位：传统奇幻卡通风，轮廓要清楚
- 色彩策略：阵营和兵种职责一眼可区分
- 单位优先级：前排更厚重，后排更轻巧，Boss 明显更大
- Excel 壳层：延续工作簿、表格框线、状态栏、页签等元素

### 10.2 音效方向

- 出兵音效：轻快、明确
- 技能音效：明显高一级，形成节奏节点
- Boss 登场：专属低频提示
- 主堡受击：必须有警报感
- 胜利/失败：短促直接，不拖沓

## 11. 技术与实现难点

### 11.1 核心难点

- 单线战场里同时表现前中后排和空层
- 保证大量单位同屏时仍然清楚可读
- 让高费单位和技能真正改变战线，而不是只改数字
- 关卡脚本既有主题差异，又不需要巨量特例逻辑

### 11.2 渲染架构：分层 Canvas

采用 4 层 Canvas 分离渲染，避免重复绘制：

```ts
const layers = {
  background: createCanvas(),  // 背景（静态，只画一次）
  ground: createCanvas(),      // 地面/路径（静态）
  units: createCanvas(),       // 单位（动态，每帧重绘）
  effects: createCanvas(),     // 特效/血条（瞬时）
};

function render() {
  ctx.drawImage(layers.background, 0, 0);
  ctx.drawImage(layers.ground, 0, 0);
  ctx.drawImage(layers.units, 0, 0);
  ctx.drawImage(layers.effects, 0, 0);
}
```

### 11.3 性能优化方案

#### 11.3.1 批量渲染（Batch Rendering）

按单位类型分组批量绘制，减少 Canvas 状态切换：

```ts
function renderUnits(units) {
  // 同颜色单位批量画
  ctx.fillStyle = '#FF4444';
  units.filter(u => u.type === 'melee').forEach(drawUnit);
  
  ctx.fillStyle = '#4444FF';
  units.filter(u => u.type === 'ranged').forEach(drawUnit);
  
  ctx.fillStyle = '#AA44FF';
  units.filter(u => u.type === 'mage').forEach(drawUnit);
}
```

#### 11.3.2 LOD 降级（Level of Detail）

根据单位在屏幕上的大小决定绘制细节：

```ts
function drawUnit(unit) {
  const screenX = unit.x - camera.x;
  
  // 屏幕外不画
  if (screenX < -50 || screenX > canvas.width + 50) return;
  
  // 根据距离决定细节
  if (unit.scale > 0.8) {
    drawDetailedUnit(unit);  // 完整火柴人 + 动画
  } else if (unit.scale > 0.4) {
    drawSimpleUnit(unit);    // 简化火柴人
  } else {
    drawDotUnit(unit);       // 一个点 + 颜色
  }
}
```

#### 11.3.3 固定时间步长 + 逻辑降频

逻辑更新和 AI 更新降频，渲染保持 60 FPS：

```ts
const LOGIC_DT = 1000 / 30;  // 逻辑 30 FPS
const AI_DT = 200;           // AI 每 200ms 更新（5 FPS）
let logicAccumulator = 0;
let aiAccumulator = 0;

function gameLoop(delta) {
  logicAccumulator += delta;
  aiAccumulator += delta;
  
  // 逻辑更新（30 FPS）
  while (logicAccumulator >= LOGIC_DT) {
    updateMovement(LOGIC_DT);
    updateCombat(LOGIC_DT);
    logicAccumulator -= LOGIC_DT;
  }
  
  // AI 更新（5 FPS）
  if (aiAccumulator >= AI_DT) {
    updateAllAI();
    aiAccumulator = 0;
  }
  
  render(); // 渲染 60 FPS
}
```

#### 11.3.4 空间分区（网格系统）

把战场分成网格，碰撞检测只查相邻网格，从 O(n²) 降到 O(n)：

```ts
const GRID_SIZE = 40;
const grid = new Map();

function updateGrid(units) {
  grid.clear();
  units.forEach(unit => {
    const key = `${Math.floor(unit.x / GRID_SIZE)},${Math.floor(unit.y / GRID_SIZE)}`;
    if (!grid.has(key)) grid.set(key, []);
    grid.get(key).push(unit);
  });
}

// 只检测相邻 9 个网格
function getNearbyUnits(unit) {
  const gx = Math.floor(unit.x / GRID_SIZE);
  const gy = Math.floor(unit.y / GRID_SIZE);
  const nearby = [];
  
  for (let dx = -1; dx <= 1; dx++) {
    for (let dy = -1; dy <= 1; dy++) {
      const key = `${gx + dx},${gy + dy}`;
      if (grid.has(key)) nearby.push(...grid.get(key));
    }
  }
  return nearby;
}
```

#### 11.3.5 对象池（Object Pooling）

预分配单位对象，避免频繁创建/销毁导致 GC 卡顿：

```ts
// 预分配 500 个单位对象
const unitPool = Array(500).fill(null).map((_, i) => ({
  id: i, active: false, x: 0, y: 0, hp: 0, type: '',
}));

function spawnUnit(type, x, y) {
  const unit = unitPool.find(u => !u.active);
  if (!unit) return null;
  unit.active = true;
  unit.type = type;
  unit.x = x; unit.y = y;
  return unit;
}

function removeUnit(unit) {
  unit.active = false; // 归还对象池
}
```

#### 11.3.6 事件驱动战斗

不用每帧检测，用事件触发状态变化：

```ts
class CombatSystem {
  attack(attacker, target) {
    const damage = calculateDamage(attacker, target);
    target.hp -= damage;
    if (target.hp <= 0) this.emit('unitDeath', target);
  }
}

combat.on('unitDeath', (unit) => {
  handleDeath(unit);
  spawnLoot(unit);
});
```

## 15. 测试要点

### 15.1 核心规则测试

- 出兵费用与冷却是否正确
- 单位交战、死亡、推进是否正确
- 空军与地面单位是否按规则结算

### 15.2 内容测试

- 每关是否有清晰主题，不会变成纯数值堆叠
- 兵种克制是否成立，不会出现单一兵种通杀
- 技能是否真的能救线或开团，而不是鸡肋按钮
- 英雄成长是否有感知，不会升了等于没升

### 15.3 体验测试

- 玩家是否能在第一章内迅速学会“补前排 / 补对空 / 憋大怪”
- 单局时长是否稳定在 5~8 分钟
- 失败后是否能快速重开并理解为什么输

## 17. 最终设计结论

这套玩法最适合被定义为：

**“传统奇幻题材包装的轻量赤潮式 PvE 战线对推游戏”**

它最重要的设计判断有 5 条：

1. 不做重 PvP，改做章节 PvE
2. 不做重编队，改做轻配队
3. 局内操作以出兵为主，技能为关键节奏点
4. 英雄负责局外成长，不负责场上手操
5. 单局保持 5~8 分钟，优先保证清晰、爽快、可读

如果后续进入实现阶段，这份设计可以直接拆成：

- 单位与战场运行时
- 兵种内容表
- 英雄与技能表
- 关卡脚本表
- UI 壳层与结果流转

这样既符合项目当前的 arcade 方向，也保留了“赤潮式横版战线”最值得留下的那部分乐趣。
