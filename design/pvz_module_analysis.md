# PvZ 模块正式分析稿

- 日期：2026-04-12
- 模块：`Sheet7 / Sheet8 / Sheet9`
- 分析框架：`game-studio -> web-game-foundations -> game-ui-frontend`
- 当前结论：PvZ 已经是“可玩的第一版”，下一阶段不该继续零散补单位，而要切到“章节化内容生产 + 战斗规则补完 + UI 分层稳态”路线

---

## 1. 模块定位

### 1.1 在工作簿合辑里的角色

PvZ 是当前合辑里最接近“中局策略主玩法”的模块。

- Aim：极短局、热手、即时反馈
- Snake：短局、风险累积、再来一把
- Tetris：中局、整理控场、稳定心流
- PvZ：中局、资源经营、波次防守、章节扩展
- Perler：长局、创作、完成感

PvZ 的职责不是“再补一个小游戏”，而是承接整套合辑里最适合做持续更新的内容模块。

### 1.2 目标玩家感受

PvZ 的核心感受应当是：

- 先做卡组选择，再进入布防
- 资源紧、格子少、波次推进有压迫
- 通过布阵和卡组理解获得稳定优势
- 打完一局后想换卡组、换章节、换规则再来一次

---

## 2. `game-studio` 视角：玩法总判断

### 2.1 当前玩法形态

从现有实现看，PvZ 现在已经具备完整闭环的最低版本：

- 有 5x9 棋盘
- 有选卡阶段
- 有阳光经济
- 有植物放置
- 有基础僵尸推进
- 有弹道与胜负判定
- 有图鉴页和章节/实验室页

这说明 PvZ 不再处于“是否能做出来”的阶段，而是进入“怎样做成可持续迭代模块”的阶段。

### 2.2 当前最明显的问题

当前 PvZ 的问题不是不能玩，而是三个层面混在一起：

- 内容层已经扩了植物和僵尸名录，但规则层还没有完全跟上
- 章节层已经有概念，但主玩法并没有真正按章节切换规则
- UI 层已经有 Sheet7~9 拆分，但战斗内信息层级仍偏薄

这会导致玩家感觉“看起来内容很多，但实际策略差异还不够大”。

### 2.3 模块下一阶段目标

PvZ 下一阶段应该从“可玩第一版”切到“章节化策略模块第一版”。

目标不是继续单点加内容，而是把这三件事连起来：

1. 章节决定规则
2. 卡组决定打法
3. 图鉴/实验室为主战斗提供回流

## 3. `web-game-foundations` 视角：架构边界

### 3.1 现在的边界状态

当前 PvZ 已经有正确雏形：

- `pvzBoardState.ts` 负责主状态推进
- `PvZGameSheet.tsx` 负责 tick 和 UI 编排
- `pvzPlantRegistry.ts` / `pvzZombieRegistry.ts` 承担内容定义
- `pvzChapters.ts` 独立存放章节概念

这个方向是对的，但还不够彻底。

### 3.2 现在最需要补的架构层

PvZ 下一阶段最需要补的是 4 个边界。

#### 第一层：章节规则边界

现在章节定义在 `pvzChapters.ts`，但没有成为真正的战斗输入。

正式版应该让章节成为主状态构建参数，而不是只出现在说明页。

建议抽出：

```ts
type PvZScenarioDefinition = {
  id: string;
  chapterId: string;
  allowedPlants: PvZPlantId[];
  zombieWaves: PvZSpawnEvent[];
  economyRule: 'day' | 'night' | 'pool' | 'fog' | 'roof';
  boardModifiers: string[];
};
```

这样主战斗才能真正由“章节配置”驱动。

#### 第二层：植物规则边界

当前注册表里已经有不少植物字段，但部分植物效果还没真正落地。

从现状看，以下植物/机制明显还处于“名录先行、规则未补完”状态：

- `cherryBomb`
- `potatoMine` 的埋伏/起爆完整流程
- `threepeater` 的三路逻辑
- `torchwood` 的弹道强化逻辑
- `kernelPult` 的投掷差异

所以后续不能继续只往 registry 里加名字，必须先建立“植物能力执行器”边界。

#### 第三层：波次脚本边界

现在刷怪队列是硬编码线性时间轴，够用，但不适合做章节和关卡。

建议升级成“场景波次脚本”：

- 预热波
- 压力波
- 精英波
- 收尾波

每一波能描述：

- 什么时候来
- 哪几路来
- 是否联动特殊规则

#### 第四层：存档/进度边界

PvZ 现在更像临时运行态，没有正式的模块进度结构。

下一阶段至少要定义：

- 已解锁章节
- 已发现单位
- 最近卡组
- 最近战斗结果
- 可恢复战斗

### 3.3 推荐的正式模块边界

建议 PvZ 模块收敛成下面这层结构：

```text
PvZ Module
├─ scenario definitions
├─ unit registries
├─ pure battle simulation
├─ battle selectors
├─ battle ui
├─ collection ui
├─ lab ui
└─ storage/progression
```

这样后面加章节、加单位、加新规则时，不会继续挤进 `PvZGameSheet.tsx`。

## 4. `game-ui-frontend` 视角：界面层分析

### 4.1 当前 UI 方向是否成立

现在的 PvZ UI 方向是成立的：

- 战斗页单独在 `Sheet7`
- 图鉴页单独在 `Sheet8`
- 章节/规则页单独在 `Sheet9`

这非常符合工作簿壳层，不需要推翻。

### 4.2 当前 UI 的主要短板

#### 短板 1：战斗页信息层级偏薄

现在局内能看到：

- HUD
- 卡槽
- 棋盘
- 选卡提示

但还缺少更明确的“当前战斗目标”和“波次压力提示”。

玩家能玩，但不够像成品。

#### 短板 2：选卡阶段的策略信息不够

现在选卡更多是“能选”，而不是“为什么选这套”。

正式版应该让选卡阶段回答这几个问题：

- 这一关主要压力是什么
- 这套卡怎么解
- 当前卡组还缺什么

#### 短板 3：战斗后回流不够强

赢了或输了之后，现在更像结束状态，而不是内容回流节点。

正式版应提供：

- 再来一局
- 更换卡组
- 查看本章图鉴
- 进入实验室看规则说明

### 4.3 PvZ 推荐 UI 结构

PvZ 战斗页建议固定成 4 层：

1. 顶部短状态带
   - 当前章节
   - 波次进度
   - 防线状态

2. 选卡/卡槽层
   - 选卡阶段和战斗阶段共用同一位置

3. 主棋盘
   - 永远是视觉中心

4. 底部或侧边辅助信息
   - 当前提示
   - 本关重点威胁
   - 快捷说明

图鉴和实验室则继续留在独立 Sheet，不要塞回战斗页。

## 5. 当前版本能力评估

### 5.1 已经成立的能力

- 战斗最小闭环成立
- 选卡阶段成立
- 单位库已形成扩展方向
- 图鉴与实验室页已拆出
- Excel 包装和模块位置成立

### 5.2 当前阻止它继续升级的关键问题

- 章节还不是战斗真实输入
- 多个植物只是“已命名”，还不是“已实现策略单位”
- 波次逻辑还不适合做多关卡生产
- 战斗内提示和复盘不足
- 还没有正式的 PvZ 模块进度层

---

## 6. 下一阶段功能包

### 6.1 功能包 A：章节驱动战斗

这是 PvZ 最优先的更新包。

目标：

- `Sheet9` 不再只是说明页
- 章节配置真正进入 `Sheet7`
- 不同章节有不同经济、波次和单位池

完成后效果：

- 白天、夜晚、泳池、迷雾、屋顶会真正打出差异
- 图鉴和实验室内容有实际用途

### 6.2 功能包 B：植物能力补完

优先补“名录已存在但规则未完整”的植物。

建议顺序：

1. `potatoMine`
2. `cherryBomb`
3. `threepeater`
4. `torchwood`
5. `kernelPult`

这是比继续加新植物更重要的事情。

### 6.3 功能包 C：战斗结算与回流

补成品感最明显的一包。

包括：

- 胜利/失败结算层
- 本关总结
- 推荐下一步
- 快速去图鉴/实验室/重开

### 6.4 功能包 D：PvZ 模块进度

包括：

- 已通章节
- 已发现植物/僵尸
- 最近卡组
- 章节最佳战绩

这会把 PvZ 从“单局模块”变成“长期模块”。

---

## 7. 建议的实施顺序

不要按“单位列表”更新，建议按下面顺序做：

1. 章节驱动战斗
2. 植物能力补完
3. 战斗结算与回流
4. PvZ 模块存档/进度
5. 新章节与新单位扩充

这样做的好处是，后面每加一个植物或僵尸，都能挂在稳定框架里，而不是重新整理结构。

---

## 8. 当前正式判断

PvZ 现在已经值得作为重点模块继续做，但更新策略必须切换。

接下来不该再按“多放几个植物、再补几个僵尸”推进，而应按下面一句话执行：

**先把 PvZ 做成章节驱动的可持续模块，再做内容扩容。**
