# 音乐节奏游戏（Rhythm Game）Excel 游戏详细设计

- 日期：2026-04-14
- 项目：`excel-aim-trainer`
- 设计目标：在现有 Excel 风格游戏合辑中新增下落式音游模块，提供"节奏感训练 + 手指协调 + 心流体验"的 arcade 玩法
- 设计阶段：实施前详细设计

---

## 1. 游戏概述

### 1.1 一句话介绍

一个伪装成 Excel 数据表格的下落式音乐节奏游戏，用 DFJK 四键敲击下落的音符，在老板眼皮底下练反应和节奏感。

### 1.2 核心卖点

- **节奏感训练**：通过精确的判定窗口训练玩家的节奏感知和手指协调，与 FPS 练枪的"反应速度"形成互补
- **心流体验**：音乐 + 节奏 + 连击的三重反馈，提供与现有模块完全不同的沉浸感
- **Excel 伪装**：音符是"数据行"，判定线是"汇总行"，连击是"处理进度"，完美融入工作簿语境
- **低门槛高上限**：4 键入门简单，但精确判定和高速谱面提供长期挑战空间
- **短局循环**：每首曲目 1~3 分钟，适合碎片时间反复练习

## 2. 原版游戏分析

### 2.1 下落式音游核心机制

下落式音游（VSRG, Vertical Scrolling Rhythm Game）的核心循环：

```
音符从上方沿轨道下落 → 到达判定线时按下对应按键 → 系统根据时间差给出判定 → 得分 + 连击
```

#### 经典游戏参考

| 游戏 | 键位数 | 核心特色 | 可借鉴点 |
|------|--------|----------|----------|
| OSU! Mania | 4~10 键 | 社区谱面生态、多难度 | 判定窗口分级、谱面编辑器思路 |
| Deemo | 单轨钢琴 | 叙事驱动、钢琴曲库 | 判定精度、长按音符设计 |
| 节奏大师 | 4~5 键 | 流行曲库、滑动音符 | 滑动判定、连击特效 |
| Beatmania IIDX | 7 键 | 街机音游标杆 | 判定严格度、Groove Gauge |
| Cytus | 扫描线 | 创新判定方式 | 视觉反馈设计 |

### 2.2 判定系统设计（行业通用标准）

下落式音游的判定核心是**时间窗口**——玩家按键时刻与音符到达判定线时刻的时间差。

| 判定等级 | 时间窗口（±ms） | 得分比例 | 连击效果 | 视觉反馈 |
|----------|-----------------|----------|----------|----------|
| Perfect / Marvelous | ±16ms | 100% | 连击 +1 | 金色闪光 |
| Great | ±35ms | 80% | 连击 +1 | 蓝色闪光 |
| Good | ±70ms | 50% | 连击 +1 | 绿色闪光 |
| Miss | > ±70ms | 0% | 连击中断 | 灰色/红色 |

> 注：不同游戏窗口略有差异。OSU! Mania 的 Perfect 约 ±16ms，节奏大师约 ±25ms。本项目首发采用中等严格度，后续可通过难度档位调节。

### 2.3 谱面设计原则

1. **节奏对齐**：每个音符必须与音乐节拍严格对齐，不能偏离 BPM 网格
2. **难度递进**：从单键 → 双键 → 多键 → 复杂节奏型，逐步提升
3. **可读性**：音符间距不能过密，玩家需有足够反应时间
4. **音乐表达**：谱面应反映音乐的情绪变化，高潮段落密度增加
5. **公平性**：不放置"不可读"或"反人类"的音符排列

### 2.4 用户界面布局（经典下落式）

```
┌─────────────────────────────────────┐
│  歌曲信息 | 分数 | 连击 | 准确率     │  ← 顶部 HUD
├─────────────────────────────────────┤
│                                     │
│    ↓    ↓    ↓    ↓    ← 音符下落区  │
│    █    █         █                 │
│         █    █                      │
│    ↓    ↓    ↓    ↓                 │
│                                     │
├─────────────────────────────────────┤
│  [D]  [F]  [J]  [K]   ← 判定线 + 键位│
├─────────────────────────────────────┤
│  进度条 | 暂停 | 重开               │  ← 底部控制
└─────────────────────────────────────┘
```

## 3. 核心玩法规则

### 3.1 音符下落

- 音符从屏幕顶部沿固定轨道匀速下落
- 下落速度由 BPM 和难度共同决定
- 音符必须在正确的节拍时刻到达判定线

### 3.2 判定线

- 固定在屏幕下方约 1/4 处
- 视觉上伪装成 Excel 的"汇总行"或"分隔线"
- 4 条轨道对应 D、F、J、K 四个按键

### 3.3 按键时机

- 当音符中心与判定线重合时按下对应按键
- 系统记录按键时刻与理论时刻的时间差
- 根据时间差给出 Perfect / Great / Good / Miss 判定

### 3.4 得分计算

```
单音符得分 = 基础分 × 判定倍率 × 连击加成
```

- 基础分：100 分
- 判定倍率：Perfect = 1.0, Great = 0.8, Good = 0.5, Miss = 0
- 连击加成：见第 7 节数值平衡表

### 3.5 结算评级

| 评级 | 条件 | 显示 |
|------|------|------|
| S+ | 准确率 ≥ 98% 且全连 | `=评级: S+ 完美演奏` |
| S | 准确率 ≥ 95% | `=评级: S 优秀` |
| A | 准确率 ≥ 90% | `=评级: A 良好` |
| B | 准确率 ≥ 80% | `=评级: B 合格` |
| C | 准确率 ≥ 70% | `=评级: C 需练习` |
| D | 准确率 < 70% | `=评级: D 继续加油` |

准确率 = (Perfect × 100% + Great × 80% + Good × 50%) / 总音符数

## 4. 判定系统设计

### 4.1 判定窗口（精确时间）

首发采用**中等严格度**，适合大众玩家，后续通过难度档位调节：

| 难度档位 | Perfect (±ms) | Great (±ms) | Good (±ms) | Miss (>±ms) |
|----------|---------------|-------------|------------|-------------|
| 简单 | ±30 | ±60 | ±100 | > ±100 |
| 普通 | ±22 | ±45 | ±80 | > ±80 |
| 困难 | ±16 | ±35 | ±65 | > ±65 |
| 专家 | ±12 | ±25 | ±50 | > ±50 |

### 4.2 判定流程

```
1. 音符到达判定线附近（进入 Good 窗口）
2. 玩家按下对应按键
3. 计算 timeDiff = 按键时刻 - 理论到达时刻
4. 根据 timeDiff 查表得出判定等级
5. 更新分数、连击、准确率
6. 触发对应视觉/音效反馈
7. 若 Miss，连击归零
```

### 4.3 连击系统

- 每次 Perfect / Great / Good 判定，连击数 +1
- Miss 判定，连击归零
- 连击数影响得分倍率（见第 7 节）
- 连击达到里程碑时触发特殊提示

#### 连击里程碑

| 连击数 | 提示文案（公式栏风格） |
|--------|----------------------|
| 10 | `=连续处理 10 行数据` |
| 25 | `=批量处理效率提升中` |
| 50 | `=数据流稳定运行 50 次` |
| 100 | `=百行无差错处理！` |
| 200 | `=超大规模数据处理中` |
| 500 | `=系统性能极限测试` |

### 4.4 准确率追踪

实时显示当前准确率，让玩家了解自己的表现：

```
准确率 = Σ(每个音符的判定得分) / (总音符数 × 100) × 100%
```

- Perfect = 100%
- Great = 80%
- Good = 50%
- Miss = 0%

## 5. 音符类型设计

### 5.1 单点音符（Tap Note）

- **表现**：单个方块沿轨道下落
- **操作**：在音符到达判定线时按下对应按键
- **Excel 包装**：像一条待处理的数据记录
- **占比**：占所有音符的 60~80%

### 5.2 长按音符（Hold Note）

- **表现**：长条形音符，有头部和尾部
- **操作**：头部到达时按住按键，直到尾部通过判定线
- **判定**：
  - 头部判定：与单点音符相同
  - 持续判定：按住期间每帧检查，松开则后续部分判 Miss
  - 尾部判定：松开时刻的精度判定
- **Excel 包装**：像一段需要持续处理的数据范围
- **得分**：按住期间持续得分，完整释放有额外奖励

### 5.3 滑动音符（Slide Note）

- **表现**：带方向箭头的音符，从一个轨道滑向另一个轨道
- **操作**：在起始轨道按下按键后，快速滑向目标轨道
- **判定**：起始点判定 + 滑动方向判定
- **Excel 包装**：像数据从一个单元格拖拽到另一个单元格
- **出现频率**：中等难度以上开始出现

### 5.4 连打音符（Jack Note）

- **表现**：同一轨道上密集排列的多个单点音符
- **操作**：快速连续按同一按键
- **判定**：每个音符独立判定
- **Excel 包装**：像批量处理同一列的多条数据
- **出现频率**：困难难度以上，用于制造高潮段落的紧张感

### 5.5 音符类型汇总表

| 类型 | 首发 | 普通难度 | 困难难度 | 专家难度 |
|------|------|----------|----------|----------|
| 单点 Tap | ✅ | ✅ | ✅ | ✅ |
| 长按 Hold | ✅ | ✅ | ✅ | ✅ |
| 滑动 Slide | ❌ | ✅ | ✅ | ✅ |
| 连打 Jack | ❌ | ❌ | ✅ | ✅ |

## 6. 谱面系统设计

### 6.1 BPM 同步机制

所有音符基于 BPM（每分钟节拍数）网格生成：

```
每拍间隔(ms) = 60000 / BPM
```

- 音符只能放置在拍点或拍点的分数位置（1/2、1/4、1/8 拍）
- 不允许偏离网格的音符
- 变速曲目支持 BPM 变化事件

### 6.2 难度分级

每首曲目提供 3~4 个难度等级：

| 难度 | 标签 | 音符密度 | 适合人群 |
|------|------|----------|----------|
| Easy | `=难度: 入门` | 低，以单键为主 | 新手，熟悉操作 |
| Normal | `=难度: 标准` | 中，加入双键和长按 | 有一定节奏感的玩家 |
| Hard | `=难度: 进阶` | 高，复杂节奏型 | 熟练玩家 |
| Expert | `=难度: 挑战` | 极高，连打和滑动 | 音游老手 |

### 6.4 谱面数据结构

```ts

  durationMs: number;
  bpm: number;
  audioUrl: string;
  charts: RhythmChart[];

};
```

## 7. 数值平衡表

### 7.1 判定窗口总表

| 难度 | Perfect | Great | Good | Miss | 适用场景 |
|------|---------|-------|------|------|----------|
| 简单 | ±30ms | ±60ms | ±100ms | > ±100ms | 新手入门 |
| 普通 | ±22ms | ±45ms | ±80ms | > ±80ms | 日常练习 |
| 困难 | ±16ms | ±35ms | ±65ms | > ±65ms | 进阶挑战 |
| 专家 | ±12ms | ±25ms | ±50ms | > ±50ms | 极限精度 |

### 7.2 得分倍率表

| 判定 | 基础倍率 | 说明 |
|------|----------|------|
| Perfect | 1.0x | 满分 |
| Great | 0.8x | 小幅扣分 |
| Good | 0.5x | 明显扣分 |
| Miss | 0x | 不得分 |

### 7.3 连击加成表

| 连击区间 | 加成倍率 | 累计效果 |
|----------|----------|----------|
| 0~9 | 1.0x | 无加成 |
| 10~24 | 1.1x | +10% |
| 25~49 | 1.2x | +20% |
| 50~99 | 1.3x | +30% |
| 100~199 | 1.4x | +40% |
| 200~499 | 1.5x | +50% |
| 500+ | 1.6x | +60% |

### 7.4 长按音符得分

```
长按总分 = 头部判定分 + (持续帧数 × 每帧得分) + 尾部判定分
```

- 头部判定分：与单点音符相同
- 每帧得分：基础分 × 0.01（按住期间每帧获得）
- 尾部判定分：与单点音符相同
- 中途松开：后续部分全部判 Miss，已得分保留

### 7.5 结算分数计算

```
总分 = Σ(单音符得分 × 连击加成)
准确率 = Σ(判定得分) / (总音符数 × 100) × 100%
最大连击 = 本局最高连续命中数
```

## 8. 操作流程

### 8.1 键盘映射

| 键位 | 功能 | 轨道 |
|------|------|------|
| `D` | 第 1 轨道按键 | 左 1 |
| `F` | 第 2 轨道按键 | 左 2 |
| `J` | 第 3 轨道按键 | 右 1 |
| `K` | 第 4 轨道按键 | 右 2 |
| `Space` | 开始 / 重开 | - |
| `P` | 暂停 / 继续 | - |
| `Esc` | 退出到选曲 / 触发工作簿隐藏 | - |
| `1`~`4` | 数字键备用映射 | 对应 4 轨道 |

### 8.2 操作状态机

```
idle --(Space/选曲确认)--> countdown
countdown --(3-2-1 倒数结束)--> playing
playing --(P)--> paused
paused --(P)--> playing
playing --(曲目结束)--> result
playing --(Esc)--> confirm_exit
confirm_exit --(确认)--> song_select
result --(Space)--> song_select
result --(R)--> replay_same_song
```

### 8.3 选曲流程

```
1. 进入 Sheet（音游页）
2. 显示曲目列表（Excel 表格样式）
3. 用 ↑↓ 键选择曲目，←→ 键选择难度
4. 按 Space 确认，进入 3-2-1 倒数
5. 倒数结束，音乐开始，音符下落
```

### 8.4 暂停菜单

暂停时显示覆盖层，提供选项：

- `继续`（P / Space）
- `重开本曲`（R）
- `返回选曲`（Esc）
- `设置`（S）：调节下落速度、音效音量

## 12. 技术实现难点分析

### 12.1 音频同步（核心难点）

**问题**：浏览器音频播放存在延迟，音符时刻必须与音乐严格同步。

**解决方案**：

1. 使用 Web Audio API 的 `AudioContext.currentTime` 作为时间基准
2. 音符时刻基于音频时间戳，而非系统时间
3. 预加载音频，避免加载延迟
4. 使用 `requestAnimationFrame` 驱动渲染循环，每帧检查音符状态

```ts
// 时间同步核心逻辑
const audioContext = new AudioContext();
let audioStartTime: number; // 音乐开始播放的 audioContext.currentTime

function getAudioElapsedMs(): number {
  return (audioContext.currentTime - audioStartTime) * 1000;
}

// 每帧检查需要判定的音符
function tick() {
  const elapsed = getAudioElapsedMs();
  // 找出当前时刻需要判定的音符
  const activeNotes = notes.filter(
    n => n.timeMs - elapsed <= judgmentWindow && n.timeMs - elapsed >= -judgmentWindow
  );
  // ...
}
```

### 12.2 判定精度

**问题**：JavaScript 的事件循环和渲染帧率不稳定，可能影响判定精度。

**解决方案**：

1. 按键事件记录精确时间戳（`performance.now()`）
2. 判定计算基于音频时间，而非渲染帧
3. 使用 `performance.now()` 而非 `Date.now()`（精度更高）
4. 按键事件与音符判定的时间差控制在 1ms 以内

### 12.3 性能优化

**问题**：高速谱面可能同时存在大量音符，DOM 渲染可能卡顿。

**解决方案**：

1. **Canvas 渲染**：音符下落区域使用 Canvas 渲染，而非 DOM
2. **对象池**：音符对象复用，避免频繁创建/销毁
3. **视口裁剪**：只渲染屏幕内的音符，屏幕外的不渲染
4. **节流更新**：HUD 更新频率限制在 30fps，音符渲染 60fps

### 12.4 渲染方案选择

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| 纯 DOM | 与 Excel 风格一致 | 高速时性能差 | ❌ |
| DOM + Canvas 混合 | 轨道用 Canvas，HUD 用 DOM | 需要两层同步 | ✅ 推荐 |
| 纯 Canvas | 性能最好 | 与 Excel 风格割裂 | ❌ |

**推荐方案**：轨道区域用 Canvas 渲染音符，HUD 和选曲页用 DOM/React 组件。

### 12.5 音频预加载

- 曲目音频在选曲确认后开始预加载
- 显示加载进度条
- 加载完成后进入倒数
- 加载失败显示错误提示并提供重试

### 12.6 跨浏览器兼容

- Web Audio API 在主流浏览器中支持良好
- `performance.now()` 精度在各浏览器中一致
- 需要处理 Safari 的自动播放策略（需要用户交互后才能播放音频）

## 18. 当前结论

下落式音游作为 arcade 模块的新成员，其"节奏感训练 + 手指协调"的核心体验与现有模块形成互补：

- **核心机制清晰**（音符下落 → 按键判定 → 得分连击），状态机简单
- **与现有模块的"状态层 + UI 壳层"模式完全兼容**
- **单局时长可控**（1~3 分钟），与现有模块切换节奏一致
- **谱面数据驱动**，易于扩展和调优
- **Canvas 渲染轨道 + DOM 渲染 HUD 的混合方案成熟可行**
- **高可扩展**：同一运行时可扩展更多曲目、难度、音符类型

建议按本设计推进，优先完成 `Phase A`，先让 `Sheet16` 跑通基础循环，再补音效、完整谱面和存档。
