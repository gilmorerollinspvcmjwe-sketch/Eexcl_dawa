# 贪吃蛇模块正式分析稿

- 日期：2026-04-12
- 模块：`Sheet10`
- 分析框架：`game-studio -> web-game-foundations -> game-ui-frontend`
- 当前结论：贪吃蛇已经是完整短局模块雏形，下一阶段重点不是继续加特效，而是补“模式差异、记录成长、事件层”和“正式模块存档”

---

## 1. 模块定位

### 1.1 在工作簿合辑里的角色

贪吃蛇是当前合辑里最适合承担“高频短局循环”的 arcade 模块。

它和其它模块的分工很明确：

- Aim：反应与命中
- Snake：路径规划与风险累积
- Tetris：结构整理与控场
- PvZ：资源经营与章节策略

Snake 的价值不在“经典复刻”，而在于它最适合做成工作日里随时开一把的短局模块。

### 1.2 目标玩家感受

玩家进入 Snake 后应该感受到：

- 看一眼就懂
- 立即开始
- 越玩越紧
- 死了马上想重开

如果做不到这 4 件事，再多花样都没有意义。

---

## 2. `game-studio` 视角：玩法总判断

### 2.1 当前玩法形态

Snake 现在已经具备稳定的最小闭环：

- 有 classic / timed / challenge
- 有难度
- 有基础食物和特殊食物
- 有障碍
- 有开始、暂停、重开、退出
- 有公式栏与状态栏接入

这说明它已经脱离 demo 状态，进入“怎么把短局做得更耐玩”的阶段。

### 2.2 当前最明显的问题

Snake 当前最大的问题不是不能玩，而是“局内变化层还偏薄”。

玩家现在主要只在做：

- 移动
- 吃东西
- 避障
- 死亡

这套循环是成立的，但还不够支撑成品化留存。

### 2.3 当前需要从什么方向升级

Snake 后续应该从“稳定能玩”升级到“有目标层的短局模块”。

重点不是继续铺更大的地图，而是补这 3 层：

1. 模式差异更明确
2. 局内事件更有变化
3. 局外记录与成长更完整

---

## 3. `web-game-foundations` 视角：架构边界

### 3.1 当前边界状态

Snake 目前的基础边界是健康的：

- `snakeBoardState.ts` 负责纯状态推进
- `snakeSelectors.ts` 负责文案与摘要
- `SnakeSheet.tsx` 负责 tick、输入绑定、与工作簿通信
- `SnakeHud` / `SnakeOverlay` / `SnakeBoard` 分工清楚

这意味着它非常适合继续扩，而不需要先重做结构。

### 3.2 下一阶段最需要补的边界

#### 第一层：模块存档边界

现在 Snake 更像运行态模块，而不是正式持久化模块。

后续至少要定义：

- 最佳分数
- 最佳长度
- 今日局数
- 模式最佳成绩
- 最近一局可恢复状态

否则 Hub 很难真正把它当成“正式模块”管理。

#### 第二层：模式规则边界

现在 3 个模式已经存在，但差异仍然偏轻。

建议把模式从“参数变化”升级成“规则变化”：

- `classic`：稳定成长
- `timed`：限时冲分
- `challenge`：障碍与事件主导

这样后面再加活动模式时，也有统一扩展位。

#### 第三层：事件系统边界

当前特殊食物已经有了第一步，但“事件”还没有正式成为系统。

建议抽出：

```ts
type SnakeRunModifier = {
  id: string;
  type: 'speed_up' | 'meeting_block' | 'hazard_spawn' | 'score_bonus';
  durationMs?: number;
};
```

后续会议遮挡、异常格、短时奖励就不会继续散落在主循环里。

#### 第四层：成长与任务边界

Snake 非常适合做轻成长：

- 解锁模式
- 特殊食物图鉴
- 每日挑战
- 历史记录

这部分建议单独成为 `snakeProgression`，不要塞进 UI 组件。

### 3.3 推荐的正式模块边界

```text
Snake Module
├─ pure board simulation
├─ mode config
├─ food registry
├─ run modifiers / events
├─ selectors
├─ ui surfaces
└─ storage / progression
```

## 4. `game-ui-frontend` 视角：界面层分析

### 4.1 当前 UI 方向是否成立

Snake 当前 UI 方向是正确的：

- 棋盘居中
- HUD 在右
- 覆盖层处理开始/暂停/死亡
- 公式栏与状态栏同步

这已经符合工作簿壳层，不需要推翻。

### 4.2 当前 UI 的主要短板

#### 短板 1：模式差异感在 UI 上不够强

虽然已经有模式切换，但玩家切进不同模式时，页面的目标提示变化还不够明显。

正式版应该让玩家一眼知道：

- 这一局是活得久
- 还是抢时间
- 还是过障碍

#### 短板 2：结算层不够“成品”

现在有结算覆盖层，但还不够承接复玩。

正式版结算层应明确给出：

- 本局成绩
- 是否破纪录
- 死亡原因
- 再来一局
- 返回主页

#### 短板 3：局内变化提示偏弱

特殊食物和加速存在，但玩家不一定足够明确感知“局势变化了”。

Snake 这种模块非常依赖清晰提示，因为局时短、节奏快。

### 4.3 Snake 推荐 UI 结构

Snake 正式版建议固定为 4 个持续面：

1. 棋盘
2. 右侧核心 HUD
3. 底部或侧边图例/提示
4. 开始、暂停、结算覆盖层

不建议继续加很多常驻小面板，避免把短局模块做成配置页。

## 5. 当前版本能力评估

### 5.1 已经成立的能力

- 核心循环稳定
- 输入与状态边界清楚
- 工作簿集成已完成
- 文案和 UI 已接近正式模块
- 适合继续做轻成长与事件扩展

### 5.2 当前阻止它继续升级的关键问题

- 模式差异仍偏参数化
- 事件系统还没正式成层
- 没有正式模块级存档与成长
- 结算与记录回流偏轻
- Hub 还没有吃到真实历史数据

---

## 6. 下一阶段功能包

### 6.1 功能包 A：模式差异强化

这是 Snake 最优先的更新包。

建议目标：

- `classic`：追求最长链和高分
- `timed`：追求限时效率
- `challenge`：地图事件和障碍带来更强变化

模式差异一旦明确，Snake 的复玩性会立刻上升。

### 6.2 功能包 B：结算与记录

包括：

- 最佳分数
- 最佳长度
- 今日局数
- 模式最佳记录
- 破纪录提示

这是让 Snake 进入“成品模块”的最低门槛。

### 6.3 功能包 C：事件层

建议优先做轻量事件，不要一上来做复杂脚本。

优先级建议：

1. 咖啡强化提示
2. 短时奖励格
3. 临时障碍生成
4. 会议遮挡事件

### 6.4 功能包 D：Snake 成长层

包括：

- 模式解锁
- 特殊食物收集
- 每日挑战
- 跨模块成就

这会让 Snake 从“随手玩一把”升级成“会回来看记录”的模块。

---

## 7. 建议的实施顺序

Snake 不需要先推翻现有结构，建议按下面顺序推进：

1. 模式差异强化
2. 结算与记录
3. 事件层
4. 成长与解锁
5. 主题和高级彩蛋

这样既能保住当前稳定性，也能最快补足成品感。

---

## 8. 当前正式判断

Snake 已经具备成为正式模块的基础，但还缺最后一层产品化包装。

接下来应按一句话执行：

**先把 Snake 变成有目标、有记录、有变化的短局模块，再去扩更多花样。**
