# 黄金矿工乱码修复与 Canvas 素材重画设计

## 目标

本次只处理黄金矿工相关内容，完成两件事：

1. 修正黄金矿工前台可见乱码，统一回到 UTF-8 正常显示。
2. 将黄金矿工战场里的可见物件改成独立 Canvas 素材方案，后续其他游戏也沿用这套做法。

本次不扩主线玩法，不新增新模式，只收口显示质量和素材结构。

## 设计原则

- 整体仍保持 Excel 壳层风格，不改工作簿外框和主导航基调。
- 矿区里的钩子、炸药、金块等物件改得更像真实游戏素材，但不能脱离当前项目的“表格伪装”语境。
- 页面组件只负责摆放和调用素材，不再把每个物件的绘制细节直接塞进主棋盘渲染逻辑。
- 素材代码必须独立存放，便于后续别的游戏复用同类结构。

## 范围

### 本次纳入

- 黄金矿工 Sheet、图鉴页、窗口标题里的相关乱码修复
- 战场 Canvas 中以下内容重画：
  - 钩子

  - 金块

  - 石头
  - 钱袋

  - 蝙蝠
  - 高亮/抓取相关视觉反馈

### 本次不纳入

- 新关卡、新模式、新商店道具
- 音效系统补齐
- 黄金矿工以外模块的乱码清理
- 跨游戏统一素材注册中心

## 方案对比

### 方案 A：继续把素材画法写在 GoldMinerBoard 里

- 优点：改动最少
- 缺点：文件会继续变重，后面别的素材也会继续堆进来

### 方案 B：按职责拆成多个 Canvas 素材文件（本次采用）

- 优点：结构清楚，适合黄金矿工继续扩，也方便后面别的游戏照着做
- 缺点：会多出几份文件，需要顺手整理调用关系

### 方案 C：直接上全项目通用素材框架

- 优点：长期最统一
- 缺点：这轮范围会明显变大，容易把一次显示修复做成全项目重构

## 最终方案

采用方案 B。

黄金矿工新增独立素材目录，把战场素材按类别拆开，`GoldMinerBoard` 只做：

- 画背景和网格
- 根据状态决定画哪个物件
- 调用素材绘制函数
- 处理基础布局和层级

每个素材文件只负责“一个类别怎么画”，不负责业务状态推进。

## 文件结构

建议新增：

```text
src/features/gold_miner/assets/
├─ drawGoldMinerHook.ts       // 钩子、绳索、抓爪
├─ drawGoldMinerLoot.ts       // 金块、钻石、石头、钱袋、神秘袋
├─ drawGoldMinerCreatures.ts  // 鼹鼠、蝙蝠
├─ drawGoldMinerEffects.ts    // 高亮、锁定、炸药、抓取反馈
└─ index.ts                   // 统一导出
```

现有主要改动文件：

- `src/components/gold_miner/GoldMinerBoard.tsx`
- `src/App.tsx`
- `CONTEXT.md`

如有黄金矿工图鉴页需要同步素材文案，也在 `GoldMinerGuideSheet.tsx` 内小幅调整。

## 绘制规则

### 钩子

- 钩绳保持当前“工具/设备”感
- 钩头改成更明确的机械抓爪轮廓
- 回收时抓到物品的挂接关系要更明显

### 战利品

- 金块按大小画出体积感和金属高光
- 钻石要和金块一眼区分开，偏冷色透明感
- 石头要明确低价值感，避免继续像普通圆点
- 钱袋和神秘袋保留袋状轮廓和封口区分

### 生物类

- 鼹鼠偏地面目标，强调移动目标身份
- 蝙蝠偏空中目标，轮廓更轻更尖

### 效果类

- 高亮效果继续服务玩法，不做花哨特效
- 炸药表现要让玩家一眼看出“已炸毁目标”
- 被探测器标记的物件要比普通描边更清楚

## 数据流

运行时状态仍然完全来自现有 `goldMinerBoardState.ts`。

本次只增加“渲染层解释”：

1. `GoldMinerBoard` 从 state 读取物件种类、坐标、是否高亮、是否被抓取。
2. 按物件种类分派到对应素材绘制函数。
3. 素材函数只消费绘制参数，不直接读取 React 状态。

这样可以保证玩法逻辑和视觉实现继续分离。

## 乱码修复策略

- 先只修黄金矿工相关前台可见乱码
- 重点检查：
  - `App.tsx` 中黄金矿工窗口标题
  - 黄金矿工 Sheet 和 Guide 页相关文案
  - 如有黄金矿工专属状态文案乱码，一并修正
- 修改时统一按 UTF-8 读写，不扩大到别的模块

## 错误处理

- 素材绘制函数如果遇到未知物件种类，回退到简单占位图形，避免整块战场空白
- 统一把默认分支保留在渲染层，避免新物件接入时直接炸掉画布
- 乱码修复只改确定有问题的黄金矿工可见文本，不碰无关文件编码

## 对后续游戏的约束

本次会把以下结论同步进 `CONTEXT.md`：

- 后续新游戏素材默认采用“独立文件 + Canvas 绘制函数”的方案
- 页面组件不再直接内联大段素材绘制细节
- 优先按“道具 / 单位 / 特效 / 交互部件”拆文件
